# 量化项目落地全景计划

## 0. 范围与约束

- **基线**：`docs/量化项目构建方案分析.md`（跨市场多模态智能量化交易系统）。
- **资源前提**（用户确认）：单人/兼职、本地资源为主，先做能跑通的最小闭环；LLM、知识图谱、VeighNa 等重型组件先用**可替代的轻量方案**落地，验证闭环后再升级。
- **不写时间/日期**：阶段只用编号 + 依赖关系；每阶段以"可独立验收"为准。
- **优先级**：A 股 ETF / 行业板块 + 长线基本面 + 事件驱动新闻情绪闭环 → 之后再扩美股与短线撮合。

## 2. 差距与替代策略（轻量化优先）

| 方案重型组件 | 落地替代（Phase 1-4）| 升级路径（Phase 5+） |
|---|---|---|
| Neo4j + LlamaIndex GraphRAG | NetworkX 图 + SQLite 持久化 + 关键词/规则 NER | Neo4j Aura Free / Memgraph + LlamaIndex |
| GPT-4-Vision 解析 PDF | `pdfplumber` + 正则 + 财报标准表头模板 | 本地 Qwen-VL / 云端 GPT-4o-mini |
| Multi-Agent Debate | 单 LLM + ROLES Prompt + 规则后置校验（数字一致性、来源回链）；可选 DeepSeek API | 多 Agent 框架（AutoGen / LangGraph） |
| FinBERT / Financial LLM | SnowNLP + 自建金融情感词典 + 关键词加权 | HuggingFace `yiyanghkust/finbert-tone` 本地推理 |
| GNN 传导 | 图上 BFS + 衰减权重传播（已有 taxonomy） | PyTorch-Geometric / DGL |
| VeighNa 实盘 | 现有 `execution/paper.py` + `broker.py` mock/replay；保留接口 | vnpy_xtp / vnpy_ib 真实柜台 |
| Airflow | `scripts/daily_pipeline.py` + Windows 计划任务 / cron | Airflow / Prefect |
| RL 中央代理 | 规则化 Regime Gating（VIX/回撤/情绪三档切权重） | Stable-Baselines3 PPO |
| Backtrader / Vectorbt / Qlib | 沿用 `backtest/engine.py` + V2 风险 | 引入 vectorbt 做大规模因子并行 |

### Phase 1 — 知识底座 llmwiki MVP（轻量 GraphRAG）🟡 进行中

- **目标**：把方案"一"落到本地文件 + SQLite + NetworkX，可被 LLM/规则双通道检索。
- **现状对照**
  - 已有：`knowledge/graph.py`（仅 dict 构建函数，无持久化、无 NER、无检索）；`knowledge/taxonomy.py` 4 个 seed 行业；`data_store` 有 news_items/policy_items/financial_reports/source_documents 表，**无 knowledge_nodes/edges 表**。
  - 待补：图持久化、规则 NER、混合检索、增量构建脚本、llmwiki 目录与 schema。

- **子任务（建议按依赖顺序）**

  - **P1.1 llmwiki 目录与 schema 文档**
    - 新建 `llmwiki/{raw,wiki}/.gitkeep`
    - 写 `llmwiki/CLAUDE.md`：说明 raw（不可变原文 + 来源 hash）vs wiki（AI 可写沉淀 + frontmatter）的写入规则；定义节点四类（industry / segment / company / policy）和边类型（has_segment / leader / supplier_of / mentioned_in / affected_by）。
    - 写 `llmwiki/README.md`：人类入口。

  - **P1.2 data_store schema 扩展**
    - `data_store/schema.py` 增表：
      - `knowledge_nodes(node_id TEXT PK, type TEXT, name TEXT, attrs_json TEXT, updated_at TEXT)`
      - `knowledge_edges(src TEXT, dst TEXT, type TEXT, weight REAL, evidence_json TEXT, updated_at TEXT, PRIMARY KEY(src, dst, type))`
      - `knowledge_evidence(node_id TEXT, doc_source TEXT, doc_hash TEXT, snippet TEXT, ts TEXT)` 索引到证据回链
    - 测试：表创建幂等、PK 冲突走 INSERT OR REPLACE。

  - **P1.3 `knowledge/graph.py` 重构为 `IndustryGraph`**
    - 类化：`IndustryGraph(NetworkX DiGraph)` + `upsert_node / upsert_edge / neighbors(node, hops=1, edge_types=None)`。
    - 持久化：`load_from_store(conn)` / `save_to_store(conn)`。
    - **向后兼容**：保留 `build_industry_graph(taxonomy, leaders) -> dict` 旧接口（内部走新类导出 dict）。
    - 测试：与旧函数输出 dict 等价；增量 upsert 不丢历史属性。

  - **P1.4 `knowledge/extractors.py`：规则 NER**
    - 公司抽取：基于 `data_store` 已知 symbol→name 词典 + akshare 股票列表缓存（fallback 到 `data/cache/stock_list.csv`）。
    - 产业链关键词：从 `DEFAULT_TAXONOMY` 拍平成 alias 词典。
    - 政策标签：维护少量"十五五/产业政策/反垄断/出口管制"等模式正则。
    - API：`extract(text: str) -> ExtractionResult{companies, segments, policies}`。
    - 测试：覆盖中英文混排、别名命中、负样本不假阳。

  - **P1.5 `knowledge/retrieval.py`：HybridRetriever**
    - `BM25Retriever`（自实现，scikit 替代避免引外部包；语料 = news_items + policy_items + 行业卡片摘要）。
    - `GraphNeighborRetriever`：给定查询里的实体节点，取 k 跳邻居作为候选文档来源。
    - `HybridRetriever.search(query, top_k) -> [Hit{doc_id, score, source, evidence_chain}]`：RRF 融合 BM25 + 图邻居召回。
    - 预留 `VectorRetriever` 抽象接口（Phase 7 接向量库）。
    - 测试：RRF 行为对齐公式；空结果回退；evidence_chain 一定有回链。

  - **P1.6 `scripts/build_knowledge_graph.py`**
    - 入口：读 `data_store` 中 news_items / policy_items / financial_reports → 调 extractor → upsert IndustryGraph + knowledge_evidence → save。
    - 幂等：基于 (src, dst, type) PK；重复跑只更 updated_at + 累加 weight（cap）。
    - 接入 `scripts/daily_pipeline.py` 作为 `update_knowledge` 后的新 step（feature flag 关闭默认）。
    - 测试：tmp DB + 假新闻 → 节点/边数符合预期；二次跑无新增节点。

  - **P1.7 文档 + 配置**
    - `config.yaml` 新增 `knowledge: { graph: { enabled: true, neighbor_hops: 2 }, retrieval: { top_k: 10, rrf_k: 60 } }`。
    - `utils.config.cfg` expose `cfg.knowledge_*` 便捷读取。
    - `docs/architecture.md` 在"目标架构"图下加一段"Phase 1 实际落点"。

- **依赖**：Phase 0 ✅。
- **验收**
  - `python -m pytest knowledge/ tests/ -q` 全绿（新增 ≥ 8 个用例）。
  - `python scripts/build_knowledge_graph.py` 在样例 DB 上跑通，落库节点数 > 0、边数 > 0、证据可回查。
  - 给定一条新闻调 `HybridRetriever.search`，返回 top-k 含图邻居命中且带 evidence_chain。
  - Phase 1 末做 schema review（plan §5 R3 风险闸门）。

### Phase 2 — 基本面摄取与防幻觉框架

- **目标**：方案"二"的本地化版本；先 A 股巨潮 PDF，后接 SEC EDGAR。
- **交付物**

  - `data/filings/parser.py`：`pdfplumber` 抽取三大表 + 关键科目映射；落到 `data_store.financial_reports`。
  - `knowledge/prompts.py`：ROLES 框架模板（Role/Objective/Limits/Expectations/Safeguards），同时输出"无 LLM 也能跑"的规则版护城河打分。
  - `knowledge/validator.py`：MAD 的单 LLM 替代——数字一致性校验、来源回链强制、置信度阈值，不通过则降级到规则。

### Phase 4 — 策略层：长线 Alpha + 事件驱动 + 规则化调度

- **目标**：方案"五"的最小化落地。

## 4. 横切关注点

- **配置**：所有新模块必须通过 `utils.config.cfg` 读取，禁止硬编码标的/路径；`config.yaml` 增 `knowledge / filings / sentiment / portfolio` 段。
- **幂等性**：所有摄取脚本沿用 `INSERT OR IGNORE` + 内容 hash 模式；新增表写迁移脚本，禁止手改 schema。
- **测试**：每阶段交付物**必须**带 `tests/` 覆盖，至少覆盖 happy path + 一个失败回退路径；Phase 0 完成后 `python -m pytest` 必须始终绿。
- **可观测性**：所有 pipeline step 落 `pipeline_runs`；关键数值（覆盖率、命中率、回测指标）写入 `data_store`，由 dashboard 消费。
- **合规与风控**：`execution/live_guard` 是任何实盘类改动的"门"，禁止绕过；`risk_controls` 阈值在 `config.yaml` 集中。
- **中文优先**：注释、CLI 输出、文档继续中文。
- **Skill 流程**：功能型改动先用 `brainstorming` → `writing-plans` → TDD → `verification-before-completion`。

## 5. 风险与决策点

- **R1 LLM 依赖**：MAD/抽取若依赖外部 API，需在 `config.yaml` 配开关，离线场景必须降级到规则版。决策点：Phase 2 完成后评估是否引入 DeepSeek API。
- **R2 PDF 解析鲁棒性**：巨潮 / SEC 表格格式多变，`parser.py` 必须有"解析失败 → 落原文 + 人工标注队列"兜底。
- **R3 图谱 schema 早期不稳**：Phase 1 的节点/边模型一旦写入持久化，后续迁移代价大；Phase 1 末必须做 schema review checkpoint。
- **R4 实盘风险**：Phase 5 之前禁止任何真实下单代码合入 `main`；VeighNa 适配器必须默认 dry-run。
- **R5 单人维护**：每阶段交付物拆到"半天可完成"的子任务（见 SQL todos），避免一次性大改。
